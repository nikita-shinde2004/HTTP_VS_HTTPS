💀 Common Attacks on HTTP

Unencrypted HTTP is vulnerable to several well-known attacks. HTTPS mitigates all of these.

1. Man-in-the-Middle (MITM)

An attacker positions themselves between the client and server, intercepting all communication. With HTTP, they can read and even modify the data in transit. 

Client ──── [ATTACKER intercepts + reads/modifies] ──── Server
      
Example: Attacker sees your bank credentials in plain text.
         Attacker can change "transfer $10" to "transfer $10000".
HTTP: VULNERABLE  |  HTTPS: PROTECTED ✅

2. Packet Sniffing / Eavesdropping

Tools like Wireshark can capture all network traffic on a local network. Over HTTP, every request — including passwords, session tokens, and private messages — is visible. 

$ wireshark -i eth0

Captured Packet:
  HTTP POST /login
  username=nikita&password=mypassword123  ← EXPOSED ⚠️
HTTP: VULNERABLE  |  HTTPS: PROTECTED ✅

3. Session Hijacking

After you log in, a session cookie is issued. Over HTTP, this cookie is transmitted in plain text and can be stolen by an attacker to impersonate you without needing your password. 

HTTP Request Headers (intercepted):
  Cookie: session_id=abc123xyz  ← Attacker steals this!

Attacker replays:
  GET /dashboard HTTP/1.1
  Cookie: session_id=abc123xyz  ← Now impersonating you!
HTTP: VULNERABLE  |  HTTPS: PROTECTED ✅

4. SSL Stripping

An attacker downgrades a connection from HTTPS to HTTP. The user thinks they are on HTTPS, but the attacker intercepts an unencrypted version. HSTS (HTTP Strict Transport Security) prevents this attack. 

User types: https://bank.com
Attacker: "I'll downgrade that to http://bank.com"
User gets: http://bank.com  ← Unencrypted without knowing!

Defense: HSTS Header forces HTTPS always.
HTTP: VULNERABLE  |  HTTPS + HSTS: PROTECTED ✅
